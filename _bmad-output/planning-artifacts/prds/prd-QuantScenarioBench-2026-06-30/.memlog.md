- (decision) Core product per brief: finance-scoped JAX framework for market scenario simulation via common state-space API (randomness, latent_state, observation, metadata); v1 models Black-Scholes/Heston/rBergomi; HF benchmark dataset publishing is primary deliverable built on framework; primary success criterion is API stability under extension; oracle labels (theoretical pricing, hedging) out of v1 scope, acknowledged as long-term direction via AD-through-paths
- (decision by user) v1 interface is Python API only, no CLI, until clear need emerges. Usage shape: simulate(model=BlackScholes(...), time_grid=..., n_paths=..., seed=...) returning a Scenario object
- (decision by user) Scenario stable top-level schema: observation, latent_state, metadata. randomness exists in the interface conceptually but materialized randomness is optional (not always returned). Reproducibility guaranteed via metadata (seed/PRNG info), not via returned raw randomness
- (decision by user) latent_state is model-specific: e.g. Heston exposes variance process, Black-Scholes may expose no latent state at all
- (decision by user) Model parameters are typed dataclasses implemented as JAX PyTrees, not plain dicts
- (decision by user) v1 HF datasets published per model (one dataset per model), sharing the same top-level schema across models. Versioned datasets are a core v1 deliverable
- (decision by user) Dataset file format: Parquet. Library and published datasets are versioned independently (separate version schemes)
- (decision by user) time_grid is an explicit TimeGrid object (not start/stop/steps tuple), to naturally support non-uniform grids
- (decision by user) v1 scoped to single-asset models only (no multi-asset/basket support)
- (decision by user) Model parameter validation: validate where practical, but emit warnings rather than hard errors for research-specific constraints (e.g. Heston Feller condition) - allows researchers to deliberately explore constraint-violating parameter regions
- (decision by user) Stakes calibration: launch-grade design quality/rigor (public OSS project) but incremental/intentionally small initial implementation - PRD should be solid enough for public scrutiny while v1 build stays focused
