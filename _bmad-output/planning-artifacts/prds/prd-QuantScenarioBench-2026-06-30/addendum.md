# Addendum: QuantScenarioBench PRD

Technical-foundation context carried forward from the [Product Brief addendum](../../briefs/brief-QuantScenarioBench-2026-06-30/addendum.md) that the PRD deliberately does not resolve (capabilities, not implementation — see PRD §0 Document Purpose). Intended as direct input to the Architecture phase, not as a constraint this PRD imposes.

## SDE / path-integration foundation

The brief's addendum identified `diffrax` (Patrick Kidger) as the most likely foundation layer for QuantScenarioBench's actual path simulation: it is JAX-native, autodiff- and GPU-capable, and already solves the numerical-integration problem (Stochastic Runge-Kutta methods, adjoint-based backprop) — but has no opinion on finance-domain modeling. QuantScenarioBench's value, per that reasoning, is the layer diffrax doesn't have: named Market Models, the state-space schema (PRD §3 Glossary), and the dataset export path (PRD Feature 4.4) — not reimplementing SDE integration.

This PRD treats "build on diffrax" as the leading candidate, not a commitment (PRD Open Question 8). Whatever the Architecture phase decides, it should produce the same observable behavior described by the PRD's FRs (FR-7 through FR-9, FR-4's reproducibility guarantee) — the PRD's requirements do not assume any particular solver internally.

## Why not tf-quant-finance as a base

`tf-quant-finance` (Google) already has a comparable "common API across stochastic processes" pattern (`ItoProcess` base class). It was considered and rejected as a foundation during the brief conversation: it is TensorFlow-based, not JAX, and its architecture is tied to TensorFlow idioms rather than being a thin wrapper swap. This is why FR-7 and FR-8's correctness checks specify an independently implemented reference rather than reuse of a general-purpose quant library's pricing formulas as a bundled dependency — the project's stated bet is its own integration of model + schema + dataset pipeline, not assembly from an existing common-API library.

## Candidate approach for rBergomi ground-truth checks

FR-9 leaves the rBergomi statistical/distributional test suite as an open design point (`[ASSUMPTION]`). The brief's addendum names a specific candidate worth Architecture's attention: Markovian approximations of rough Bergomi (e.g. aBergomi), which trade some model fidelity for tractable, faster-to-compute reference values usable as a Monte Carlo ground truth in tests — distinct from the long-term AD-through-paths direction for *oracle labels* (which is a published-dataset concern, out of v1 scope per the PRD), this is purely about what the v1 test suite checks simulated paths against.

## Benchmark layer: technical context for Architecture (added 2026-07-02)

The PRD's Feature 4.5–4.8 (Metrics, Traditional Baselines, Portfolio Optimizer Interface, Benchmark Runner) deliberately leave several implementation choices to the Architecture phase, consistent with the PRD's capabilities-not-implementation stance (§0). Context carried forward from this update's discovery conversation:

- **GMV and CVaR Optimization need a numerical solver.** Global Minimum Variance has a closed-form solution via covariance-matrix inversion for the unconstrained case, but a constrained variant (e.g. long-only) needs a QP solver. CVaR Optimization is a linear program in its standard (Rockafellar–Uryasev) formulation. Architecture should decide whether these are hand-rolled (e.g. via `jax.numpy.linalg` for GMV's closed form, or a JAX-native optimizer) or delegate to an external solver dependency (e.g. `cvxpy`, `scipy.optimize`) — the latter would be the project's first non-JAX-native optimization dependency, worth flagging against the framework's JAX-native identity (Cross-Cutting NFRs).
- **Portfolio Optimizer Interface typing should probably mirror `MarketModel` (AD-1).** The State-Space Interface is an `equinox.Module` ABC (AD-1); for architectural consistency, `BaselineStrategy` and `ForecastOptimizer` are natural candidates for the same pattern (project-wide `equinox` pytree convention, AD-6) rather than a plain Python Protocol — but this is an Architecture-phase call, not a PRD commitment.
- **Rebalancing and window-sourcing are confirmed (PRD FR-27, Open Questions 12–13, resolved 2026-07-02).** A strategy is fit once per Benchmark Runner call, on a lookback window preceding the evaluated window (both drawn from the same Scenario), and its Portfolio Weights are held statically (buy-and-hold) over the full evaluated window — no refitting at points along the `TimeGrid` in v1. This is the simplest option consistent with v1's narrow-scope posture elsewhere; Architecture should build against it directly rather than treat it as still open.
- **Forecasting-model integrations named as motivating examples, not v1 work.** The user named PatchTST, iTransformer, and TimeMixer as the kind of models the `ForecastOptimizer` interface (FR-24) should accommodate later. None are v1 deliverables (§6.2) — they exist here only to sanity-check that the interface shape (`historical_returns + forecast → weights`) is generic enough to wrap a learned sequence model without the Benchmark Runner needing to know about it.
- **No landscape research on existing portfolio-optimization/backtesting libraries was done for this update** (unlike the SDE-solver side, which explicitly weighed `diffrax` against `tf-quant-finance` before deferring to Architecture — see above). Libraries like PyPortfolioOpt, Riskfolio-Lib, and empyrical/quantstats implement GMV/CVaR optimization and standard performance metrics already; Architecture should decide whether any of these are worth reusing for the solver/metric internals (Open Question 10) versus a from-scratch JAX-native implementation, the same build-vs-adopt tradeoff already resolved once for the SDE side.
- **Return-series derivation (FR-28) is a genuine open gap, not just a solver detail.** How a Scenario's `observation` (a price path) becomes the return series that Metrics (Feature 4.5) and strategies (Feature 4.6/4.7) consume — simple/arithmetic return vs. log return, and at what sampling frequency relative to the `TimeGrid` — was not specified during this update's discovery conversation (Open Question 16). This is upstream of the solver-choice questions above and should be resolved early in Architecture, since every downstream Feature 4.5–4.8 computation depends on it.

## Evaluation Results & Leaderboard: technical context for Architecture (added 2026-07-03)

The PRD's Feature 4.9 (`EvaluationResult` schema, publishing, Leaderboard aggregation) deliberately leaves the exact schema shape and Architecture-phase mechanics open, consistent with the PRD's capabilities-not-implementation stance (§0). Two established conventions from the Hugging Face ecosystem ground the design, carried forward here as direct input to Architecture (AD-26):

- **The `model-index` results-block convention.** Hugging Face model/dataset cards support a `model-index.results` YAML front-matter block — a list of `{task, dataset, metrics}` records, where `metrics` is itself a list of `{name, type, value, verified}` entries — that the Hub auto-renders and that feeds Papers-with-code sync. This is why FR-30 fixes `EvaluationResult.metrics` as a list of `{name, value}` records rather than reusing `BenchmarkResult`'s flat `dict[str, float]` (AD-24) — the two types serve different consumers and are allowed to diverge in shape (AD-26).
- **The `open-llm-leaderboard` / `lm-evaluation-harness` results-repo convention.** That ecosystem's leaderboard is backed by a dedicated Hugging Face dataset repo accumulating one JSON file per evaluation run (organized by submitter/model), which a separate Space reads, aggregates, and ranks. QuantScenarioBench's Feature 4.9 adopts the same split: the shared Evaluation Results repo (FR-33) is the append-only source of truth; Leaderboard aggregation (FR-34) is a pure read-and-rank function over it; a hosted Space rendering that aggregation (Feature 4.10) is a later, separate phase — architecturally a thin consumer of FR-34's output, not a place where new aggregation logic should be added.
- **No landscape research was done on alternative leaderboard backends** (e.g. a database instead of a git-versioned dataset repo, or a config-driven leaderboard framework). The git-versioned-repo approach was chosen by direct analogy to the pattern above and to QuantScenarioBench's own existing Benchmark Dataset publishing path (Feature 4.4, `export/_publish.py`), not from an independent build-vs-adopt survey; Architecture should treat this as the working assumption, not a closed decision.
